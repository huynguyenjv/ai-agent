Overall Verdict

▼

The 32 endpoints in scope plus the cross-cutting posture scan together produced 11 CRITICAL and 22 HIGH findings. The repository exhibits a systemic pattern: the authorization framework (@RbacRequired + OrgCodeValidationAspect) is designed around a single path variable name (orgCode) and the contract breaks silently on every controller that uses any other name (orgId, roleId, userId), on V2 controllers that omit the annotation entirely, and on routes the permit-all yaml lists unauthenticated.

Combined Blast Radius

Any authenticated tenant member can read, modify, or delete data in any other tenant, and can grant themselves super-admin within their own tenant - without tripping any alarm (audit logging is absent on every mutation path reviewed).

Systemic Root Causes

▼

1 OrgCodeValidationAspect is path-variable-name-dependent

Evidence: OrgCodeValidationAspect.java:42 - if (pathVars == null || !pathVars.containsKey("orgCode")) return;

Impact: The aspect short-circuits for every endpoint that uses {orgId}, {userId}, {roleId}, {staffId}, {code}, or that accepts the org from a header/body. Instead of failing closed, it allows the request.

Fix: Rewrite the aspect to fail-closed. Introduce a @TenantScoped(pathVar = "...") annotation. Add ArchUnit/Checkstyle rule.

2 V2 controllers omit @RbacRequired / @HasPermission

Evidence: PermissionV2Controller.java, RolePermissionV2Controller.java, ModuleV1Controller.java, ModuleConfigsV1Controller.java

Impact: Any authenticated principal can hit permission CRUD, run /permissions/sync to rewrite the service catalog, grant any permission to any role in any org.

Fix: Class-level @HasPermission by default on every @RestController. CI check that rejects unannotated controllers.

3 Mass-assignment on every update DTO

Evidence: OrganizationUpdateRequest exposes status + parentOrganizationId; OrganizationUserUpdateRequest exposes organizationRole (incl. SUPER_ADMIN)

Impact: Self-promotion to super-admin, org-reparenting, lockout of privileged accounts.

Fix: Split DTOs per role-scope. Privileged fields on dedicated endpoints gated by dedicated permissions.

4 "Cannot grant more than you have" is nowhere enforced

Evidence: RoleUseCaseService.java:148-174, RolePermissionUseCaseServiceV2.java:51-95

Impact: An org-admin can grant any permission that exists in the catalogue to a role they hold, then reload their token and be a super-admin.

Fix: Every permission-grant path must intersect the requested set with the caller's effective permission set. Add scope metadata on PermissionEntity.

5 Audit logging is missing on every privileged mutation

Evidence: AuditLogFactory exposes only a login(...) factory. Zero matches in role, permission, organization services.

Impact: Forensic blind spot after any privilege-escalation or tenant-breach. PCI 10.2, SOC 2 CC7.3, ISO 27001 A.12.4 compliance gap.

Fix: Spring @AfterReturning aspect on every *CommandService mutation method.

6 permit-all YAML entries expose the wrong surface

Evidence: application-prd.yml:4 permits /private/** and /internal/** unauthenticated

Impact: GET /private/api/v1/profile/{userId} returns full PII unauthenticated on prd. DoS via OpenSearch re-indexing triggers.

Fix: Enumerate routes; require mTLS or client_credentials JWT on /private. Never permit /api/v1/**.

7 Tenant identity taken from client-controlled header/body

Evidence: ModuleV1Controller.java:45-62 uses X-Organization-Id header

Fix: Tenant identity must come from IdUtils.getRequiredOrgId(). Header/body org identifiers allowed only as filters.

8 Starter-delegated security with zero visibility

Evidence: No SecurityFilterChain, CorsConfigurationSource, HttpSecurity in this repo. All in microservice-spring-boot-starter:2.0.12-RELEASE.

Fix: Audit the starter separately. Add a local SecurityConfig that denies /api/** by default as defense-in-depth.

9 gt-app issuer is a universal bypass

Evidence: OrgCodeValidationAspect.java:35-36 - if ("gt-app".equals(issuer)) return;

Impact: Any JWT bearing iss=gt-app bypasses every tenant check.

Fix: Replace with scope/audience claim check. Log every invocation of the bypass path.

10 Authentication bypass via /api/v1/app/login-anonymous

Evidence: AppV1Controller.java:96 + AuthUseCaseService.java:212-238 - issues 1-year JWT to any caller supplying a valid email, no credential.

Impact: Account takeover for any user whose email is known.

Fix: Delete the endpoint. If business-justified, gate behind OTP / magic-link.

01 - Organization & Profile Endpoints

▼

Scope: /api/v1/organizations/** and /api/v1/organizations/{orgId}/profile

Critical C-01: BOLA / IDOR - cross-tenant read & modification

Affected Endpoints

  • GET /api/v1/organizations/{orgId}
  • PATCH /api/v1/organizations/{orgId}
  • POST/GET/PATCH/DELETE /api/v1/organizations/{orgId}/profile

Evidence

Controller accepts any path orgId, only validates UUID shape. Service does a blind findById with no comparison to caller's JWT orgId.

The existing tenant-check aspect applies only when the path variable is named orgCode:

// OrgCodeValidationAspect.java:42
if (pathVars == null || !pathVars.containsKey("orgCode")) return;

Impact

Tenant data exfiltration via GET. Arbitrary modification via PATCH (deactivate, re-parent). Arbitrary deletion of any tenant's profile.

Fix

UUID callerOrgId = IdUtils.getRequiredOrgId();
if (!callerOrgId.equals(orgId) && !isParentOf(callerOrgId, orgId)) {
    throw new ForbiddenException("Cross-tenant access denied");
}
Critical C-02: Mass-assignment - client can set status & parent org

Evidence

// OrganizationUpdateRequest.java:8-14
public record OrganizationUpdateRequest(
    String name,
    OrganizationType organizationType,
    OrganizationStatus organizationStatus,
    UUID parentOrganizationId
) {}

Impact

Attacker re-parents tenant B under tenant A. Self-DoS by flipping own status to INACTIVE/DELETED.

Fix

Split DTOs: self-service DTO (name only) and platform-admin DTO (status, parentOrganizationId).

Critical C-03: POST /organizations/initial has no admin gating

Impact

Unauthorized tenant provisioning. Data pollution and quota abuse. Combined with C-02 and C-01, attacker can create a tenant then re-parent existing tenants under it.

Fix

Move to platform-admin-only controller guarded by specific permission (e.g. @RequirePermission("platform.org.create")).

02 - Organization Users Endpoints

▼

Scope: /api/v1/organizations/{orgCode}/users endpoints (16-22)

Critical F-01: Cross-tenant data leak - endpoints 21 & 22 bypass tenant scoping

Evidence

OrgCodeValidationAspect.java:42 - aspect short-circuits when no {orgCode} path variable. Endpoints 21/22 have no path variable.

Impact

Any authenticated user can call GET /organizations/users and retrieve userId, fullName, email, phoneNumber of staff across every tenant. Multi-tenant breach; large-scale PII exposure.

Critical F-02: Privilege escalation via PATCH - caller can promote to SUPER_ADMIN

Evidence

Update DTO binds organizationRole (unrestricted OrgRole enum including SUPER_ADMIN), status, and roleIds.

Impact

Self-promotion to SUPER_ADMIN. Via roleIds, silently stack high-privilege roles.

High F-05: gt-app issuer blanket bypass of tenant check

Evidence

// OrgCodeValidationAspect.java:35-36
if ("gt-app".equals(issuer)) return;

Impact

Any JWT issued by gt-app skips the entire orgCode to token.orgId comparison.

03 - Permissions + Role-Permission Bindings

▼

Scope: V2 permissions CRUD, /me, matrix, sync, role-permission bindings (endpoints 23-32)

Critical F-01: V2 permission CRUD + sync gated by NO role/permission check

Evidence

PermissionV2Controller carries no @HasPermission, @PreAuthorize, @Secured, or any authZ annotation on any handler.

Endpoint 30 (/permissions/sync) is the worst: bulk-inserts/updates/soft-deletes modules and permissions by serviceCode AND invalidates every affected role's JWT claims.

Impact

Any authenticated caller can create/modify/delete system-wide permission definitions. Combined with F-02, grants arbitrary permissions to any role = full privilege escalation.

Critical F-02: PUT /roles/{roleId}/permissions - cross-tenant privilege grant

Evidence

Handler has no @HasPermission. updatePermissionByRoleId loads the role by id and rewrites its permission set without consulting the caller's organization.

Impact

A user in org A who discovers a role UUID in org B can call PUT and become a full admin of org B. Full cross-tenant privilege escalation.

High F-04: Missing audit logging on all permission mutations

Evidence

AuditLogUseCase.log(...) is called only from auth flows. Grep confirms zero call sites in any permission / role-permission service.

Impact

Forensic blind spot after a privilege-escalation incident. PCI 10.2, SOC 2 CC7.3, ISO 27001 A.12.4 compliance gap.

04 - Organization Roles Endpoints

▼

Scope: /api/v1/organizations/{orgCode}/roles (endpoints 37-42)

Critical FIND-01: BOLA / Cross-tenant permission write on endpoint 42

Evidence

updatePermissionsByOrgCode(orgCode, roleId, assign, permissionCodes) only validates org exists but never verifies that the roleId belongs to the organization identified by orgCode.

Attack

  1. Attacker is org admin of ORG_A (JWT orgId = ORG_A)
  2. They enumerate/guess a roleId belonging to ORG_B
  3. They call PATCH /api/v1/organizations/ORG_A/roles/<ORG_B_roleId>/permissions
  4. The aspect passes (URL ORG_A matches JWT). The service writes into rolepermissions against ORG_B's role.

Impact

Full cross-tenant compromise of authorization - any org admin can rewrite permissions of any role in any other org.

Critical FIND-02: Privilege escalation - no "cannot grant more than you have" check

Evidence

assignPermissions accepts any set of permission codes and writes them unconditionally. No subset check vs the caller's own effective permissions.

Impact

An org admin can grant a role (including one they hold) any permission that exists in the catalogue. If the catalogue contains any super-admin permission, they can silently self-escalate.

05 - Modules Endpoint

▼

Scope: GET /api/v1/modules/mapping + adjacent ModuleConfigs

Critical F-1: Cross-tenant IDOR - target org from client-controlled header

Evidence

getModulesByOrgId(HttpServletRequest) pulls the org id directly off X-Organization-Id header and passes it to the repository. No comparison between the header and the orgId claim on the caller's JWT.

Impact

An authenticated caller from org A enumerates the modules catalogue of any org B by sending X-Organization-Id: <B-uuid>.

High A-2: Cross-tenant config pollution on ModuleConfigs

Evidence

ModuleConfigs has no orgId field. The upsert keys a single global record per moduleId.

Impact

Any authenticated caller can overwrite global module settings, affecting every tenant consuming that module. Super-user capability at regular auth tier.
